# PM 우선순위 대시보드 시연 대본

## 1분 버전

이 화면은 개발PM이 Jira나 CSV에서 가져온 요청을 같은 기준으로 비교하고, 실행 순서를 정리하기 위한 우선순위 대시보드입니다.

핵심은 `자동 점수`와 `PM 확정`을 분리한 점입니다. 자동 점수는 영향도, 긴급도, 비즈니스 가치, 리스크, 난이도, 공수를 기준으로 추천 레인을 계산합니다. PM 확정은 모든 요청을 처음에 `Backlog`에 두고, PM이 직접 `Now`, `Next`, `Hold`로 승인하는 방식입니다.

예를 들어 결제 실패 이슈는 자동 점수 기준으로 `Now`가 추천됩니다. 하지만 실제 운영에서는 담당 팀 리소스나 릴리즈 맥락을 보고 PM이 직접 `Now 확정`, `Next 확정`, `Hold 확정`, `범위 축소` 중 하나를 선택합니다.

하단에는 공유용 요약, 담당 팀별 병목, 결정 로그가 시각화되어 있습니다. 그래서 이 화면은 단순 목록이 아니라, 요청 수집부터 우선순위 판단, 확정, 기록까지 이어지는 PM 운영 흐름을 보여줍니다.

## 3분 버전

이 대시보드는 개발PM이 기능 요청, 버그, 운영 이슈, 실험 아이디어를 한 화면에서 비교하고 실행 순서를 정리하기 위해 만든 MVP입니다.

첫 번째는 외부 소스 연결입니다. Jira 필터 영역에는 JQL이나 saved filter를 넣을 수 있고, CSV 영역에는 `title,type,owner,impact,urgency,value,risk,complexity,effort,note` 형식의 데이터를 붙여 넣을 수 있습니다. 즉, 실제 Task 관리 툴에서 가져온 데이터를 PM 판단 보드로 변환한다는 흐름을 보여줍니다.

두 번째는 검색과 필터입니다. 제목, 팀, 메모 기준으로 검색할 수 있고, Feature, Bug, Ops, Experiment 유형별 필터와 리스크 4점 이상만 보는 필터가 있습니다. 실제 업무에서는 모든 요청을 한 번에 보는 것보다, 회의 목적에 맞게 빠르게 좁혀보는 기능이 중요합니다.

세 번째는 자동 점수 보드입니다. 자동 점수 탭에서는 요청이 점수 공식에 따라 `Now`, `Next`, `Hold`로 추천됩니다. 카드에는 점수, 요청 유형, 담당 팀, 데이터 출처, 추천 레인, 판단 사유가 표시됩니다.

네 번째는 PM 확정 보드입니다. PM 확정 탭에서는 모든 요청이 처음에는 `Backlog`에 있습니다. 여기서 PM이 카드 안의 `Now`, `Next`, `Hold` 버튼을 직접 눌러 확정합니다. 확정된 요청만 해당 레인으로 이동하고, 미확정 요청은 계속 Backlog에 남습니다. 이 구조는 “점수는 추천이고 최종 판단은 PM이 한다”는 메시지를 명확하게 보여줍니다.

다섯 번째는 기록과 공유입니다. 공유용 요약은 Now, Next, Hold, Backlog 비중을 막대로 보여주고, 담당 팀별 병목은 팀별 요청 수와 고위험 항목 수를 보여줍니다. 결정 로그는 PM이 언제 어떤 요청을 어떤 레인으로 확정했는지 타임라인으로 남깁니다.

향후 실제 도입 시에는 `Generate Report` 버튼을 추가해 현재 보드 상태와 결정 로그를 Confluence에 자동 업로드하는 방식이 좋습니다. 그러면 Jira에서 Done 처리된 이슈가 보드에서 빠지더라도, 당시 PM 판단과 리포트 스냅샷은 Confluence에 기록으로 남길 수 있습니다.

## 시연 순서

1. 상단의 Source, Sprint, Last synced 정보를 보여주며 실제 운영 보드 형태라고 설명합니다.
2. 외부 소스 연결에서 Jira JQL 입력 영역과 CSV 붙여넣기 영역을 보여줍니다.
3. Jira 필터 적용 또는 CSV 가져오기로 요청이 추가되는 흐름을 설명합니다.
4. 검색과 필터에서 `Bug` 또는 `리스크 4점 이상만 보기`를 눌러 요청을 좁혀봅니다.
5. `자동 점수` 탭에서 Now, Next, Hold가 어떻게 추천되는지 보여줍니다.
6. `PM 확정` 탭으로 전환해 모든 요청이 Backlog에서 시작하는 구조를 보여줍니다.
7. Backlog 카드 안의 `Now`, `Next`, `Hold` 버튼으로 직접 확정합니다.
8. 확정된 카드가 해당 레인으로 이동하고 결정 로그가 남는 흐름을 보여줍니다.
9. 공유용 요약과 담당 팀별 병목이 확정 기준으로 바뀌는 것을 설명합니다.
10. 마지막으로 Generate Report와 Confluence 업로드 방향을 확장 아이디어로 설명합니다.

## 예상 질문 답변

**Q. 자동 점수와 PM 확정을 왜 나눴나요?**  
자동 점수는 요청을 빠르게 비교하기 위한 추천값이고, PM 확정은 일정, 팀 리소스, 릴리즈 맥락을 반영한 최종 판단입니다. 실제 업무에서는 점수만으로 결정하기보다 PM이 맥락을 보고 승인하는 구조가 더 현실적입니다.

**Q. PM 확정에서 왜 모든 요청을 Backlog에 두나요?**  
PM 확정은 자동 추천 결과를 그대로 옮기는 화면이 아니라, 사람이 직접 승인하는 단계이기 때문입니다. 그래서 미확정 요청은 모두 Backlog에 두고, PM이 직접 Now, Next, Hold로 이동시킨 요청만 확정 레인에 들어가게 했습니다.

**Q. 점수 공식은 왜 이렇게 잡았나요?**  
영향도, 긴급도, 비즈니스 가치는 실행 필요성을 높이는 요소라 가점으로 두고, 구현 난이도와 예상 공수는 일정 부담이기 때문에 차감했습니다. 리스크는 방치했을 때 손실이 커질 수 있어서 검토 우선순위를 높이는 값으로 반영했습니다.

**Q. 결정 로그는 어떤 의미인가요?**  
결정 로그는 “이 요청을 어떻게 처리하기로 했는지”를 남기는 기록입니다. Now 확정, Next 확정, Hold 확정, 범위 축소 같은 액션이 쌓이면 이후 회의나 스프린트 리뷰에서 판단 흐름을 추적할 수 있습니다.

**Q. Jira와 실제로 붙인다면 어떻게 하나요?**  
먼저 Jira JQL로 현재 스프린트의 미완료 이슈를 가져옵니다. 예를 들면 `project = GAME AND sprint in openSprints() AND statusCategory != Done`처럼 Done 항목은 제외합니다. 가져온 이슈의 제목, 타입, 우선순위, 스토리 포인트, 라벨, 담당 팀을 현재 카드 필드에 매핑합니다.

**Q. Done 이슈가 빠지면 기록이 사라지지 않나요?**  
보드에서는 Done 이슈를 제외하되, PM 결정 로그와 리포트 스냅샷은 별도로 남기는 방식이 좋습니다. 특히 Generate Report 버튼으로 Confluence에 스프린트 리포트를 업로드하면, Jira 상태가 나중에 바뀌어도 당시 판단 기록은 보존됩니다.

**Q. 이 도구의 핵심 가치는 무엇인가요?**  
요청을 많이 모으는 것이 아니라, 왜 지금 해야 하는지와 왜 보류해야 하는지를 팀이 이해할 수 있는 언어로 정리하는 것입니다. 그래서 점수, PM 확정, 공유 요약, 팀 병목, 결정 로그, 리포트화를 하나의 흐름으로 묶었습니다.

## 향후 실제 도입 방향

1. Jira JQL로 현재 스프린트의 미완료 이슈만 가져옵니다.
2. 자동 점수로 추천 레인을 계산합니다.
3. PM 확정 탭에서 모든 요청을 Backlog에 두고 직접 Now, Next, Hold로 확정합니다.
4. 결정 로그를 DB에 저장합니다.
5. Generate Report 버튼으로 현재 상태를 리포트화합니다.
6. Confluence에 스프린트별 리포트 페이지를 자동 생성합니다.
7. 리포트 URL과 생성 시각을 별도 히스토리로 저장합니다.

```text
- Now / Next / Hold / Backlog 현황
- 고위험 이슈 수
- 팀별 병목

- Issue key

- Auto lane
- Confirmed lane

- Decision reason

Decision Log
- Time
- Issue
- Action
- Score

Source
- Jira JQL
- Generated at
- Generated by
```
